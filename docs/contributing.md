# Contributing

Contributions are welcome. Whether you want to report a bug, suggest an improvement, add a feature, or fix a typo - feel free to participate.

---

## Reporting an issue

Open an issue on the GitHub repository to:

- Report a bug or unexpected behavior.
- Suggest an improvement or new feature.
- Discuss an idea before submitting a pull request.

When reporting a bug, include:

- Your browser and version
- Your Node.js version (`node -v`) if applicable
- The framework or environment (React, Vue, vanilla HTML, etc.)
- The error message and stack trace if applicable
- A minimal reproduction (CodePen, StackBlitz, or a minimal HTML file)

---

## Submitting a pull request

1. Fork the repository.
2. Create a dedicated branch:

```bash
git checkout -b my-proposal
```

3. Make your changes.
4. Build the package to verify output:

```bash
npm run build
```

5. Commit with a clear message:

```bash
git commit -m "Fix: description of the change"
```

6. Push the branch:

```bash
git push origin my-proposal
```

7. Open a pull request on GitHub.

---

## Running locally

```bash
# Install dependencies
npm install

# Build all output formats (ESM, UMD, minified, .d.ts)
npm run build
```

To test changes in a browser, open `demo/index.html` directly in your browser or serve the project directory with any static file server.

---

## Best practices

- Stay true to the focused scope of the component: theme switching via a Web Component.
- Only change what is necessary. Focused changes are easier to review.
- Test your changes in at least Chrome and Firefox. Safari fixes should be verified in Safari.
- Check that ARIA labels still update correctly after attribute changes.
- Verify that `no-shadow` mode continues to work after any template or lifecycle changes.
- Consult the [changelog](../CHANGELOG.md) to understand the history of decisions.
